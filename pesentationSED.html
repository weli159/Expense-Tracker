1. Business Context (BRD)

The Problem

Personal finance management is fragmented. Students and young professionals struggle to track daily expenses manually, leading to poor budgeting.

The Solution

A centralized Expense Tracker to record spendings, categorize data, and visualize financial health in real-time.

Objective: Reduce manual tracking time by 30%.

2. Specifications (SRS)

Functional

  • FR-1 Secure Auth (Register/Login).
  • FR-2 Expense CRUD (Add/Edit/Delete).
  • FR-3 Categorization & Types.
  • FR-4 Dashboard & Analytics.
  • FR-5 Smart Filtering (Date/Type).

Non-Functional

  • Perf Dashboard loads < 2s.
  • Sec Passwords hashed (Bcrypt).
  • Rel 99.9% Data Availability.
  • UX Responsive Mobile/Desktop.

3. Architecture & Tech Stack

We utilized a 3-Tier MERN Architecture.

  • Frontend (Presentation): React.js, React-Bootstrap, Material UI.
    Handles UI State & Visualization.
  • Backend (Application): Node.js + Express.
    REST API, Business Logic, Routing.
  • Database (Data): MongoDB Atlas.
    NoSQL Document Storage.

    Client (Browser)
       ↕ (Axios/JSON)
    Server (Express API)
       ↕ (Mongoose)
    DB (MongoDB Cloud)

Design Choices

Evolution from initial design.

  • SQL vs NoSQL:

    The initial PDF proposed MySQL. We pivoted to MongoDB because flexible JSON documents mapped perfectly to our frontend objects, speeding up development.

  • Authentication:

    We chose JWT (Stateless) over Sessions for better scalability and easier debugging.

  • API Communication:

    We used POST requests for complex filtering (instead of GET) to send secure, structured JSON bodies containing date ranges and arrays.

Technical Challenge & Resolution

The Issue: Two developers installed conflicting libraries (moment vs helmet) simultaneously on different branches.

The Result: A corrupted package-lock.json causing CI Pipeline failures and local crashes.

The Resolution (Tech Lead):

  • Halted development to prevent further corruption.
  • Manually synchronized package.json manifests.
  • Performed a "Nuclear" clean (deleted node_modules) and regenerated the lockfile.
